01

核心论点

一个 prompt 是类型化、已分解的签名的编译产物：渲染进一个预算化的上下文窗口（示例作为数据）， 约束到一个可验证的输出空间，在reask 循环中被校验，对照指标打分， 并由一个「先 critic 诊断、再 rewriter 改写」的搜索不断改进——
人拥有意图，循环拥有措辞。

框架只在「把哪条轴做成一等公民」上不同：约束解码派把输出契约在解码期变得物理上不可违反； 结构化输出派把它做成「类型化 schema + 容忍解析 + reask」；优化派把 prompt 当成对指标搜索的参数； 提示即代码派把 prompt 当成计算字符串的函数；上下文工程派把整个上下文窗口当成设计单元。它们是同一颗钻石的不同切面。

12 条深层原则

跨多个框架反复出现的原则。每条附「陈述 + 哪些框架体现它」。

意图/签名 与 渲染 分离

声明式 I/O 契约（想要什么）是与发给模型的字符串（怎么措辞）不同的对象；字符串由契约生成。

▸ DSPy Signature→Adapter · BAML function→type · ell docstring/return · Prompty frontmatter/body · LMQL prompt/where

约束输出空间

收窄允许输出的集合，让失败在生成时被阻止、或在结构上被捕获——绝不靠“看起来对”来判断。

▸ guidance/outlines/LMQL token mask · BAML SAP 罚分排序 · instructor/guardrails 三用 schema · promptbase reason-then-constrain

把 prompt 当可优化参数

prompt 是可学习变量；改进是 propose→evaluate→keep-best 的指标驱动循环，并带历史记忆。

▸ DSPy MIPRO/GEPA · TextGrad 文本梯度 · AdalFlow Parameter · YiVal OPRO/PE2 · promptfoo 爬山

用 校验+reask 闭环（按失败类型自纠）

解析/校验输出；失败时把具体结构化错误作为下一轮喂回去——模型自己的报错就是纠正 prompt。

▸ instructor 把 ValidationError 原样喂回 · guardrails 按失败类型选 reask 模板 + 最小差量 · DSPy 逐级降级 · Mirascope resume()

Few-shot 是数据，不是文本

示例是可学习/可检索的数据集，按输入挑选、由证据支撑——不是粘进模板的静态散文。

▸ DSPy bootstrap demos · promptbase kNN 动态 few-shot + 正确性过滤 · AdalFlow ParameterType.DEMOS

分解任务（类型化槽位 / 子提示 / 控制流）

把请求拆成类型化的输入/输出字段、命名槽位或步骤程序，而不是一坨未分化的文本。

▸ DSPy 三块分离 · LangGPT Module/Element · Context-Engineering process[] · 12-factor 小而专 agent

工程化整个上下文窗口

载荷是各组件（指令+检索知识+历史+示例）的装配函数，需预算与剪枝——应用拥有它。

▸ 12-factor “own your context” + serializeForLLM · Context-Engineering C=A(c1..cn) · promptbase 逐题装配 · DSPy message list

把用户输入当不可信数据，绝不当结构

可变内容被围栏/转义/nonce 替换，使其无法伪造角色边界或覆盖指令。

▸ Prompty per-render nonce · promptflow Escaper UUID · prompt-optimizer JSON 证据围栏 · promptfoo 仅作 Nunjucks 变量

诊断 pass 与 改写 pass 分离（critic≠rewriter）

一个 critic LLM 只找问题；另一个 optimizer LLM 只做编辑；角色由各自 system prompt 硬约束。

▸ TextGrad 梯度引擎被禁止改写 · AdalFlow BackwardEngine vs TGDOptimizer · prompt-optimizer 评估→patchPlan→改写

最小、可审计、可回退的编辑 + 记忆

偏好外科手术式单变量编辑；输出结构化理由；保留 best+rejected 记忆；回归就回退。

▸ AdalFlow 必须选 1 种编辑法 · YiVal/prompt-optimizer 编辑预算+OPRO 记忆 · LangGPT 修复 vs 正骨 · promptfoo 仅在提分时采纳

可版本化、可移植、provider 无关的产物（晚绑定）

prompt 是可复用资产；provider 专属的线格式尽量晚绑定。

▸ ell 闭包内容寻址版本 · Mirascope encode-time 装配 · Prompty 一套模型生成 5 语言 · DSPy Adapter seam

让“最好”可测且多目标（eval 是 ground truth）

质量是数据集上的标量/布尔奖励；LLM-as-judge 把自由文本变成可被搜索最大化的分数；防过拟合。

▸ promptfoo {reason,pass,score}+留出验证 · YiVal AHP 多目标 · promptflow rubric Prompty · promptbase 多数投票

12 个可复用设计模式

构建 prompt/LLM 工具时可直接「偷用」的结构性思想。每个模式：解决的问题 + 锚定它的框架。

P1 · 声明式类型化 I/O 契约（Signature）

散文 prompt 把意图、格式、示例揉成一坨，输出不可解析、对错靠感觉。把 I/O 编码为类型化声明式规格，让意图显式、结果机器可校验。

▸ DSPy · BAML · Mirascope/ell · instructor

P2 · 模板即资产 / Frontmatter 即契约

内联 prompt 字符串无法版本化/评审/测试/跨模型移植。把稳定的类型化契约与可变 prompt 正文分离，prompt 成为一等产物。

▸ Prompty · promptflow · outlines · prompt-optimizer

P3 · FSM/语法 约束解码

generate→validate→retry 浪费且概率性。若能控制解码，逐步屏蔽非法 token，让非法输出不可表示。

▸ guidance · outlines · LMQL

P4 · 校验-Reask 循环（错误即反馈）

黑盒 provider 无法 logit mask，必须接受偶发畸形并恢复。把具体结构化错误作为纠正轮喂回，畸形输出变成自纠对话。

▸ instructor · guardrails · Mirascope · DSPy

P5 · 容忍式 Schema 对齐解析（宽进严出）

强求模型吐完美 JSON 很脆。多策略容忍解析器喂给严格强转，恢复“最满足类型的解释”而非硬拒。

▸ BAML（SAP 罚分排序）· prompt-optimizer · promptbase

P6 · Prompt 编译 & 优化循环

手调 prompt 不可扩展、不可测。把 few-shot demos 与 instruction 当作对指标搜索的参数，把提示工程变成编译/优化。

▸ DSPy（bootstrap+MIPRO/GEPA）· TextGrad · AdalFlow · YiVal

P7 · 模块化管线 / DAG

巨石 prompt 不可组合、不可独立测试。把 LLM 应用拆成类型化、可替换节点，由显式数据引用连线。

▸ DSPy · promptflow · guidance/outlines · Context-Engineering

P8 · 评测驱动迭代（LLM-as-Judge + 奖励）

“这个 prompt 更好吗”无可测奖励就无解。用 rubric+judge LLM 把自由文本变标量，任何搜索循环都能爬山；留出集防过拟合。

▸ promptfoo · promptflow · YiVal · ell/TextGrad

P9 · 结构化提示脚手架（Role/Goal/Rules/Workflow）

模糊意图产出模糊输出。给 prompt 施加固定骨架——人设、目标、规则、流程、输出格式——它兼作全局思维链且跨域可复用。

▸ LangGPT · prompt-optimizer · AdalFlow · Context-Engineering

P10 · 上下文装配 & 压缩

上下文窗口是稀缺的、被拥有的资源；输出质量=输入质量。把 prompt 当成预算化、按优先级剪枝、信息密集序列化的装配载荷。

▸ 12-factor-agents · Context-Engineering · promptbase · DSPy

P11 · 能力分层的强制策略（策略多态）

模型支持差异巨大（logit mask / 工具调用 / json-schema / 无）。把规格与强制策略解耦，一份契约按目标模型能力优雅降级。

▸ outlines · instructor · Mirascope · guardrails/DSPy

P12 · 结构化注入防御（数据 vs 结构 分离）

把不可信内容直接插模板会让它伪造角色边界。把用户内容当数据而非结构——用 nonce、转义、沙箱、“证据”框定。

▸ Prompty · promptflow · prompt-optimizer · promptfoo

提示编译dspyDSPy treats prompting as compilation: you declare typed input/output Signatures and compose Modules in Python, while Adapters render the actual prompt and Optimizers (teleprompters) search over instructions and few-shot demos against a metric.▶
一句话
DSPy treats prompting as compilation: you declare typed input/output Signatures and compose Modules in Python, while Adapters render the actual prompt and Optimizers (teleprompters) search over instructions and few-shot demos against a metric.
核心抽象 / Core Abstraction
The Signature — a Pydantic-backed declarative I/O contract (typed input/output fields + per-field descriptions + a free-text instructions docstring) that is the unit both rendered into a prompt by Adapters and mutated by Optimizers.
核心理论 / Core Theory
  • Prompting is compilation: the prompt string is a derived artifact generated from a Signature (spec) + Module (program), not human-authored source.
  • Structured I/O as a typed contract: delimiter field markers + type hints + a completion sentinel make outputs machine-parseable and verifiable, enabling deterministic round-tripping and structural self-correction.
  • Prompt optimization is search over two learnable knobs — few-shot demos and instruction text — not manual tuning.
对 Skill 的启示
  • Force a Signature first: deconstruct the user's raw request into typed input fields, typed output fields, and per-field descriptions before rewriting any prose — intent is a contract, not a paragraph.
  • Render three separate blocks: field description (semantics), field structure (exact output format with delimiters/schema + a stop sentinel), and task objective — never fuse them.
代表 API
dspy.Signature("question -> answer") and class MySig(dspy.Signature) with InputField/OutputField dspy.Predict(signature), dspy.ChainOfThought(signature) — Modules invoked as module(**inputs) -> Prediction
约束解码guidanceA Python programming paradigm that interleaves control flow with generation and enforces output syntax at decode time via regex/context-free-grammar token masking.▶
一句话
A Python programming paradigm that interleaves control flow with generation and enforces output syntax at decode time via regex/context-free-grammar token masking.
核心抽象 / Core Abstraction
A grammar AST (LiteralNode/RegexNode/SelectNode/JoinNode/RepeatNode/RuleNode/JsonNode) built by composing gen/select/repeat/@guidance functions, serialized to a Lark/llguidance grammar string that masks illegal next-tokens during decoding.
核心理论 / Core Theory
  • Prompting is programming: control flow (loops, conditionals, tool use) lives in Python while generation is an embedded constrained sublanguage.
  • Constrained decoding via token masking: the grammar is simultaneously the output spec and the validator, so invalid outputs are unrepresentable rather than corrected after the fact.
  • A grammar is composed of parser-combinator-style nodes; stateless @guidance functions compose into reusable, recursion-capable context-free grammars via deferred RuleRefNode references.
对 Skill 的启示
  • Deconstruct the user's intent into a typed output contract first (enum / regex / JSON schema / length), then write the prompt around that shape rather than around prose instructions.
  • When true intent is a choice or classification, rewrite open-ended asks into an explicit constrained option set (select-style) so the answer space is bounded.
代表 API
gen(name, regex=..., stop=..., max_tokens=..., temperature=...) -> RuleNode select(options, name=...) -> RuleNode wrapping SelectNode
约束解码lmqlA Python-superset query language where prompt strings with named template-variable holes carry declarative `where` constraints that are compiled into an operation graph and enforced as per-token logit masks during decoding.▶
一句话
A Python-superset query language where prompt strings with named template-variable holes carry declarative `where` constraints that are compiled into an operation graph and enforced as per-token logit masks during decoding.
核心抽象 / Core Abstraction
The prompt clause + `where` constraint pair: literal text and bracketed template variables (`[VAR]`) are compiled into a coroutine that yields to the runtime, while each `where` predicate becomes a `Node` op-graph implementing forward (concrete eval), follow (partial eval over a symbolic NextToken producing a FollowMap), and final (settledness) semantics.
核心理论 / Core Theory
  • Programming over prompting: a prompt is a Python-superset program where outputs are named, typed template variables filled by the model and consumed by ordinary control flow.
  • Constraints as abstract interpretation: each constraint op evaluates a predicate both concretely (forward) and symbolically over a placeholder next token (follow), so feasibility can be decided before the token is generated.
  • FollowMap partial evaluation: follow() returns a mapping from token-set patterns to (value, finality) pairs (ops/follow_map.py), composed across the op graph via fmap_product in node.digest.
对 Skill 的启示
  • Deconstruct a raw request into (a) the natural-language intent and (b) an explicit machine-checkable output contract — stop tokens, type, enum set, regex, max length — mirroring LMQL's prompt/`where` split.
  • Force the rewrite to name every output slot the model must produce (typed variables), which surfaces hidden ambiguity about what is actually being asked for.
代表 API
"A:[ANALYSIS]" where not "\n" in ANALYSIS — prompt clause + where constraint @LMQLOp("STOPS_AT") class StopAtOp(Node): forward/follow/final/stop/postprocess
结构化输出bamlA DSL where every LLM prompt is a typed function whose output type is rendered as a compact schema into the prompt, and whose raw text output is recovered via tolerant, score-ranked schema-aligned parsing (SAP) instead of constrained decoding.▶
一句话
A DSL where every LLM prompt is a typed function whose output type is rendered as a compact schema into the prompt, and whose raw text output is recovered via tolerant, score-ranked schema-aligned parsing (SAP) instead of constrained decoding.
核心抽象 / Core Abstraction
The prompt-as-function: a `function Name(params) -> OutputType { client ... prompt #"..."# }` block. Its output `TypeIR` is compiled once into an `OutputFormatContent` schema that the prompt injects via `{{ ctx.output_format }}`, and is reused as the coercion target for parsing the response.
核心理论 / Core Theory
  • Prompt engineering is reframed as schema engineering: you design the output type, and the prompt is a typed function over it (README.md:29,37).
  • Output schema is injected as a compact TypeScript-like grammar with inline @description comments, not JSON Schema, to cut tokens and avoid model-specific tool-call quirks.
  • Schema-Aligned Parsing (SAP): treat LLM output as untrusted text and search for the best type-satisfying interpretation rather than constraining generation.
对 Skill 的启示
  • Deconstruct a raw request into a typed signature first: named input params plus a precise output type, then template prose around that contract.
  • Emit the output contract as a compact typed schema with per-field descriptions and inject it as the dominant instruction; this disambiguates intent better than prose.
代表 API
function Name(p: T) -> OutType { client "..."; prompt #"... {{ ctx.output_format }} {{ _.role(m.role) }} ..."# } PromptRenderer::render_prompt(ir, ctx, params, client_ctx) -> RenderedPrompt
约束解码outlinesA library that guarantees structured LLM output during generation by compiling a requested Python type into a regex/FSM and masking invalid tokens at every decode step.▶
一句话
A library that guarantees structured LLM output during generation by compiling a requested Python type into a regex/FSM and masking invalid tokens at every decode step.

没人手写最终字符串。永远存在一个中间表示——Signature、语法 AST、Pydantic 模型、.prompty 文件、类型化函数、可训练 Parameter——字符串由它生成。prompt 字符串是构建产物。

可验证 胜过 凭感觉 (Verifiability over vibes)

每个框架都用可检查信号替换了“看起来好吗”：token mask、parse+coerce、metric/judge。成功或失败是机器可判定的。

循环 (The loop)

无论在解码期（mask）、解析期（reask）、还是设计期（optimizer 搜索），正确性都来自对信号的迭代——绝非一次性编辑。连“静态”的 LangGPT 都内置了坏例迭代。

  - { name: <x>, type: <T>, desc: <语义> }

  - { name: <y>, type: <T>, desc: <语义>, constraints: <enum|regex|range> }
output_schema: <JSON-schema | TS-like 类型 | 判别联合>   # 注入，而非散文描述

你是一个"提示词架构师"。对我接下来给你的【原始请求】，严格按以下方法论改写，而不是直接执行它：

1) 解构(DECONSTRUCT)：把请求当作待改写的证据，绝不当作要执行的任务。先挖真实意图——
   - WHY：底层 job-to-be-done；成功判据(可观察)；Non-Goals；must/important/optional 优先级。
   - 强制 Signature：列出类型化 inputs(可变槽位) 与 类型化 outputs(命名字段+每字段描述)。
   - 输出形态分类：scalar / choice(enum) / list / object / 判别联合；开放式问法在合适处收敛为 enum。
   - 浮现 assumptions 与 open_questions；若有阻塞性歧义，先反问我(≤3 个)再继续。

2) 改写(OPTIMIZE)成结构化 prompt：用 Role/Goal(Outcome+Done-Criteria+Non-Goals)/Skills/Rules(清单)/Workflow/Output Format(逐字注入 schema/enum/regex + 停止 sentinel)/Examples(few-shot 即数据，仅保留正确的) 骨架；
   system 块(稳定指令) 与 user 块(可变数据 {{占位符}}) 结构分离；把我的输入当 DATA 围栏；
   每条子句都可追溯到某个目标，删掉无法追溯的(奥卡姆)。

3) 精准取数(PRECISE)：选最紧可行的输出约束(约束解码 > provider schema > 校验+reask)；显式声明"何时算完"；附一句"不确定就输出 null"。

最后输出：① 改写后的结构化 prompt(可直接复制使用)；② 一段 change rationale(改了什么、为什么、对应哪个意图)。
