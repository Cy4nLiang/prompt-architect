# 咨询问题库（按分型）

## 提问规则

- 先呈现自动采集摘要（一段话），再提问；
- 最多 2 轮、每轮 ≤4 问；每问必须给**推荐默认**；
- 凡能从 codebase 查到的（技术栈/命令/文件/约定）**禁止**出现在问题里；
- 用户答"你定 / 默认" → 全部取推荐项，立即停止提问。

## 第一轮：通用 4 槽位

| 槽位 | 问什么 | 推荐默认的来源 |
|---|---|---|
| 验收 | 怎么算做完？给 2–3 个候选验收口径 | 从任务描述推断的最小口径 |
| scope | 哪些不能动？（公共接口/数据格式/特定目录） | "只动 {{定位到的相关文件}}，不动公共接口" |
| 约束 | 兼容性/风格/依赖上有红线吗？ | "遵循项目现有约定，不引入新依赖" |
| 验证 | 用哪条命令验证？ | 第 1 步找到的测试/构建命令 |

## 分型替换 / 追加

| 分型 | 调整 |
|---|---|
| `feature` | + 边界场景（空输入/并发/权限…给选项）；+ 形态（API/UI/CLI 的输入输出长什么样） |
| `bugfix` | 验收槽位换成「复现步骤——我按 {{推断的步骤}} 复现，对吗？」；+ 修表象还是修根因（默认根因）；+ 要不要回归测试（默认要） |
| `refactor` | scope 槽位强化为「对外接口允许变吗」（默认不允许）；+ 现有测试能否兜底行为不变（基于采集到的覆盖情况问） |
| `perf` | + 基线与目标（当前多少 / 目标多少 / 用什么测）；+ 可接受的 tradeoff（内存换时间？牺牲可读性？） |
| `test` | + 覆盖对象优先级（哪些模块先）；+ 测试层级与 mock 约定 |

## 第二轮触发条件（满足才问，否则直接进第 4 步）

- 第一轮答案与采集事实冲突（如用户指定的验证命令实际不存在）；
- 用户自由输入引入了新的未定义实体（新文件、新接口、新外部系统）；
- 第一轮答案表明任务分型判断有误，需要改型后补对应槽位。
